## Context

当前 Cinder 系统已经实现了基于 soul 画像的对话交互和代理决策功能，但缺乏自主执行复杂任务的能力。用户需要手动将 agent 的建议转化为实际行动，这限制了系统的实用性和效率。

**现有能力**：
- Soul 画像生成和管理
- 基于 soul 的对话交互
- 代理决策（决策日志、审查）
- 多后端支持（Ollama、Claude）

**缺失能力**：
- 自主任务规划和执行
- 代码生成和文件操作
- 执行过程的反思和调整
- 端到端的任务自动化

**约束条件**：
- 必须保持向后兼容，不影响现有功能
- 必须基于现有的 soul 画像系统
- 必须使用现有的 Ollama 后端
- 必须确保文件操作的安全性
- 必须提供完整的执行日志和回滚能力

**利益相关者**：
- 终端用户：期望从目标到成果的自动化体验
- 开发者：需要清晰的架构和可扩展的设计
- 系统管理员：需要安全可控的执行环境

## Goals / Non-Goals

**Goals:**
- 实现自主任务执行引擎，接收自然语言目标并完成端到端执行
- 实现智能任务分解，将复杂目标分解为可执行的子任务序列
- 实现代码生成器，通过 Ollama 生成高质量代码
- 实现反思循环机制，基于 soul 画像评估和调整执行策略
- 实现安全的文件操作层，支持创建、修改和管理项目文件
- 实现完整的执行日志系统，记录所有决策和操作
- 提供灵活的配置管理，支持自定义执行策略和安全边界

**Non-Goals:**
- 不实现图形用户界面（GUI），仅保持命令行工具
- 不实现多用户管理或权限控制
- 不实现云端执行或分布式任务调度
- 不实现实时代码预览或热重载
- 不修改现有的 soul 画像生成逻辑
- 不修改现有的对话交互功能

## Decisions

### 1. 架构设计：模块化的执行引擎

**决定**: 采用模块化的执行引擎架构，将任务规划、代码生成、文件操作和反思机制解耦。

**架构**:
```
cinder execute <goal>
    ↓
AutonomousExecutor (协调器)
    ↓
TaskPlanner → CodeGenerator → FileOperations
    ↓              ↓               ↓
ReflectionEngine ←──────────────┘
    ↓
ExecutionLogger
```

**理由**:
- 模块化设计便于独立测试和维护
- 清晰的职责边界，每个模块专注单一功能
- 便于后续扩展和替换组件
- 支持并行执行和异步操作

**替代方案**:
- 单体架构：所有功能耦合在一起，难以维护和扩展
- 微服务架构：过于复杂，不适合 CLI 工具

### 2. 任务分解策略：基于 LLM 的层次化分解

**决定**: 使用 Ollama 模型进行任务分解，采用层次化的任务树结构。

**流程**:
```
用户目标 → LLM 分解 → 任务树 → 子任务执行 → 结果合并
                ↑____________反思循环____________|
```

**理由**:
- LLM 具备强大的语义理解和推理能力
- 层次化结构便于管理和追踪进度
- 支持动态调整和重新规划
- 可以处理模糊和开放性的目标

**替代方案**:
- 基于规则的任务分解：灵活性差，难以处理复杂场景
- 人工预设任务模板：缺乏通用性，需要大量预设

### 3. 代码生成策略：多阶段生成和验证

**决定**: 采用多阶段代码生成流程：需求分析 → 代码生成 → 格式化 → 验证 → 写入。

**流程**:
```
子任务描述 → LLM 生成代码 → Black/Ruff 格式化 → 语法检查 → 文件写入
                                ↓
                            反思评估 ← Soul 画像
```

**理由**:
- 多阶段流程确保代码质量
- 格式化和验证减少错误
- 基于 soul 的反思评估确保符合用户偏好
- 支持多种编程语言和框架

**替代方案**:
- 单次生成直接写入：错误率高，难以保证质量
- 人工审查代码：效率低，失去自动化意义

### 4. 文件操作安全：沙箱和权限控制

**决定**: 实现沙箱机制和权限控制，限制文件操作范围。

**安全机制**:
- 工作目录限制：仅允许在指定目录及其子目录操作
- 文件类型白名单：仅允许创建特定类型的文件（.py, .js, .md 等）
- 操作确认机制：高风险操作需要用户确认
- 回滚机制：支持撤销最近的文件操作
- 备份机制：修改文件前自动备份

**理由**:
- 防止意外破坏系统文件
- 避免执行恶意代码
- 提供安全可控的执行环境
- 支持错误恢复

**替代方案**:
- 无限制的文件操作：安全风险极高
- 完全禁止文件操作：失去实际意义

### 5. 反思机制：基于 Soul 的多维度评估

**决定**: 实现基于 soul 画像的多维度反思评估机制。

**评估维度**:
- **风险一致性**: 代码是否符合用户的风险偏好
- **风格一致性**: 代码风格是否符合用户的沟通偏好
- **质量评估**: 代码质量是否达到可接受标准
- **完整性检查**: 是否完成了所有子任务

**流程**:
```
执行结果 → Soul 规则匹配 → 多维度评分 → 决策建议 → 调整策略
```

**理由**:
- 基于 soul 确保个性化
- 多维度评估更全面
- 支持动态调整和优化
- 提供可解释的决策依据

**替代方案**:
- 单一维度评估：不够全面，容易遗漏问题
- 无反思机制：执行质量无法保证

### 6. 执行日志：结构化日志和可视化

**决定**: 使用 SQLite 存储结构化执行日志，支持查询和可视化。

**数据模型**:
```sql
CREATE TABLE execution_logs (
    id INTEGER PRIMARY KEY,
    timestamp TEXT,
    goal TEXT,
    task_tree TEXT,  -- JSON
    decisions TEXT,  -- JSON
    files_created TEXT,  -- JSON
    reflection_result TEXT,  -- JSON
    status TEXT
);
```

**理由**:
- SQLite 轻量级，无需额外服务
- 结构化数据便于查询和分析
- 支持执行历史回溯
- 便于后续实现可视化

**替代方案**:
- 纯文本日志：查询和分析困难
- 外部数据库：增加部署复杂度

## Risks / Trade-offs

### 风险 1: LLM 生成代码质量不稳定

**风险描述**: Ollama 模型生成的代码质量可能不稳定，存在语法错误或逻辑错误。

**缓解措施**:
- 实现代码验证和测试流程
- 提供代码审查和修正机制
- 支持回滚和重新生成
- 建立代码质量评分系统

### 风险 2: 任务分解可能不合理

**风险描述**: LLM 分解的任务可能不合理或无法执行。

**缓解措施**:
- 实现任务可行性检查
- 支持动态重新规划
- 提供人工干预接口
- 记录分解决策过程

### 风险 3: 文件操作可能破坏现有项目

**风险描述**: 自动创建或修改文件可能破坏现有项目结构。

**缓解措施**:
- 实现沙箱机制限制操作范围
- 提供备份和回滚功能
- 高风险操作需要用户确认
- 支持预览模式（dry-run）

### 风险 4: 反思机制可能增加执行时间

**风险描述**: 多轮反思循环可能显著增加任务执行时间。

**缓解措施**:
- 提供快速模式（跳过反思）
- 优化反思算法效率
- 支持并行反思评估
- 提供执行进度反馈

### 风险 5: Soul 画像可能不适用于所有场景

**风险描述**: Soul 画像可能无法覆盖所有任务场景的决策需求。

**缓解措施**:
- 提供默认决策规则
- 支持自定义决策规则
- 允许用户覆盖决策
- 记录决策过程供审查

## Migration Plan

### 阶段 1: 核心功能实现（v2.1.0）
1. 实现执行引擎核心框架
2. 实现任务分解器
3. 实现代码生成器
4. 实现文件操作层
5. 实现基本的反思机制

### 阶段 2: 功能完善（v2.2.0）
1. 实现完整的执行日志系统
2. 实现回滚和备份机制
3. 优化反思算法
4. 添加更多编程语言支持
5. 完善错误处理

### 阶段 3: 用户体验优化（v2.3.0）
1. 实现执行进度可视化
2. 提供交互式任务调整
3. 支持执行历史查询
4. 提供执行统计分析
5. 完善文档和示例

### 回滚策略
- 新功能通过独立的 `cinder execute` 命令提供，不影响现有功能
- 用户可以选择不使用新功能
- 如果出现严重问题，可以快速禁用执行引擎
- 所有文件操作都有备份，支持完全回滚

## Open Questions

1. **如何处理跨文件的代码依赖？**
   - 当前设计主要关注单文件生成，跨文件依赖可能需要额外的分析和处理
   - 是否需要实现项目级别的上下文理解？

2. **如何平衡自动化程度和用户控制？**
   - 完全自动化可能导致不可控的结果
   - 过多的人工确认会降低效率
   - 如何设计合理的确认机制？

3. **如何处理长期运行的任务？**
   - 复杂任务可能需要很长时间执行
   - 是否需要支持任务暂停和恢复？
   - 如何处理执行过程中的外部变化？

4. **如何评估代码生成的质量？**
   - 除了语法检查，是否需要运行测试？
   - 如何处理测试环境的依赖？
   - 是否需要集成 CI/CD 流程？

5. **如何处理多语言项目？**
   - 当前设计主要关注单一语言
   - 如何处理多语言混合项目？
   - 是否需要语言检测和切换机制？
