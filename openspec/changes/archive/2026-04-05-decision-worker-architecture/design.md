## Context

### 当前状态

当前 Cinder 的执行流程是线性的四阶段模型：

```
Plan → Generate → Evaluation → Decision
```

每个阶段都有独立的决策逻辑：
- **Plan**: 决定任务分解方式
- **Generate**: 决定是否继续迭代
- **Evaluation**: 决定质量是否达标
- **Decision**: 决定是否接受代码

这种设计导致：
1. 决策分散在各个阶段，缺乏全局视角
2. Decision 阶段被简化为"代码接受/拒绝"的二元决策
3. 无法处理需要人工判断的复杂场景（技术选型、架构决策等）
4. 缺乏主动交互能力，无法询问用户或调整策略

### 约束

- 必须保持向后兼容（提供迁移路径）
- 不能增加新的外部依赖
- 执行时间增加控制在 20% 以内
- 必须复用现有的 Soul profile 和决策基础设施

### 利益相关者

- **开发者**: 需要清晰的架构和易于维护的代码
- **用户**: 需要更智能的交互和更好的决策质量
- **系统**: 需要更好的扩展性和可测试性

## Goals / Non-Goals

**Goals:**

1. **实现双层 Agent Loop 架构**
   - Decision Agent 作为大脑/管理者
   - Worker Agent 作为执行者
   - 清晰的职责分离

2. **增强 Decision Agent 能力**
   - 理解用户意图和上下文
   - 基于 Soul profile 做决策
   - 主动询问用户
   - 调整执行策略

3. **简化 Worker Agent**
   - 移除所有决策逻辑
   - 专注于执行和报告
   - 返回客观数据

4. **支持复杂决策场景**
   - 技术选型
   - 架构决策
   - 多轮对话

**Non-Goals:**

1. 不引入复杂的 multi-agent 框架（如 AutoGen、CrewAI）
2. 不改变 Soul profile 的数据结构
3. 不优化 LLM 的推理速度
4. 不实现并行任务执行
5. 不支持 Agent 间的对等协作对话

## Decisions

### 决策 1: 采用双层架构而非多层 Multi-Agent

**选择**: 双层架构（Decision + Worker）

**理由**:
- 当前系统规模不需要复杂的 multi-agent
- 双层架构职责清晰，易于理解和维护
- 实现成本低，风险可控
- 为未来扩展预留空间

**备选方案**:
- 多层 Multi-Agent（Orchestrator + 多个专业 Agent）
  - 拒绝原因：复杂度过高，当前阶段不需要
- 对等 Multi-Agent（Agent 间协作对话）
  - 拒绝原因：难以预测和控制，资源消耗大

### 决策 2: Decision Agent 使用状态机模式

**选择**: 使用状态机管理 Decision Agent 的决策循环

**理由**:
- 清晰的状态转换逻辑
- 易于调试和测试
- 支持中断和恢复
- 符合"思考-决策-行动"的认知模型

**状态定义**:
```
UNDERSTAND → ANALYZE → DECIDE → DELEGATE → EVALUATE → COMPLETE
                ↑                        ↓
                └────────────────────────┘
```

**备选方案**:
- 简单的循环结构
  - 拒绝原因：缺乏明确的状态管理，难以处理复杂场景
- 使用 LangChain 的 Agent 框架
  - 拒绝原因：引入额外依赖，过度设计

### 决策 3: Worker Agent 保持当前的 Plan → Generate → Evaluation 流程

**选择**: 保持现有流程，但移除决策逻辑

**理由**:
- 最小化变更范围
- 已有的实现经过测试和验证
- 降低引入新 bug 的风险
- 渐进式重构，易于回滚

**备选方案**:
- 完全重写 Worker 流程
  - 拒绝原因：风险高，工作量大
- 将 Worker 拆分为多个专业 Agent
  - 拒绝原因：当前阶段不需要，增加复杂度

### 决策 4: Agent 间通信使用结构化消息

**选择**: 定义标准的消息格式

**消息类型**:
```python
class MessageType:
    TASK = "task"           # Decision → Worker: 执行任务
    RESULT = "result"       # Worker → Decision: 返回结果
    QUESTION = "question"   # Decision → User: 询问用户
    ANSWER = "answer"       # User → Decision: 用户回答
    OPTIONS = "options"     # Worker → Decision: 提供选项
    DECISION = "decision"   # Decision → Worker/User: 做出决策
```

**理由**:
- 类型安全，易于验证
- 支持扩展新的消息类型
- 便于日志记录和调试
- 符合消息传递模式

**备选方案**:
- 直接传递字典
  - 拒绝原因：缺乏类型安全，容易出错
- 使用事件总线
  - 拒绝原因：过度设计，增加复杂度

### 决策 5: 上下文管理使用内存 + 持久化混合模式

**选择**: 短期上下文在内存，长期上下文持久化到 SQLite

**理由**:
- 内存访问快速，适合频繁操作
- SQLite 持久化保证数据不丢失
- 支持跨会话的上下文恢复
- 复用现有的数据库基础设施

**上下文类型**:
- **短期上下文**（内存）：当前会话的对话、临时状态
- **长期上下文**（SQLite）：历史决策、用户偏好、项目信息

**备选方案**:
- 纯内存模式
  - 拒绝原因：无法跨会话恢复
- 纯持久化模式
  - 拒绝原因：性能开销大

### 决策 6: 提供向后兼容层

**选择**: 在 AutonomousExecutor 中提供兼容层

**实现**:
```python
class AutonomousExecutor:
    def execute(self, goal: str, legacy_mode: bool = False) -> dict:
        if legacy_mode:
            # 使用旧的流程和返回格式
            return self._execute_legacy(goal)
        else:
            # 使用新的 Agent 架构
            return self.decision_agent.process_goal(goal)
```

**理由**:
- 平滑迁移，不破坏现有用户
- 给用户时间适应新 API
- 便于 A/B 测试和对比

**备选方案**:
- 直接破坏性变更
  - 拒绝原因：影响现有用户
- 维护两个版本
  - 拒绝原因：维护成本高

## Risks / Trade-offs

### 风险 1: Decision Agent 可能陷入决策循环

**影响**: 无限循环，消耗资源

**缓解**:
- 设置最大决策次数（如 10 次）
- 检测重复决策，强制退出
- 记录决策历史，避免重复

**监控**: 记录每次决策循环的次数，超过阈值告警

### 风险 2: LLM 调用次数增加导致成本上升

**影响**: 执行成本增加 20-30%

**缓解**:
- 优化 prompt，减少不必要的 LLM 调用
- 使用缓存，避免重复理解相同内容
- 提供配置选项，让用户选择决策深度

**监控**: 追踪每次执行的 LLM 调用次数和 token 消耗

### 风险 3: Worker 输出格式变更导致兼容性问题

**影响**: 现有代码可能无法处理新格式

**缓解**:
- 提供格式转换工具
- 在文档中明确说明变更
- 提供迁移示例

**监控**: 收集用户反馈，及时修复问题

### 风险 4: Decision Agent 的决策质量依赖 Soul profile 准确性

**影响**: Soul profile 不准确时，决策可能不符合用户期望

**缓解**:
- 提供决策解释，让用户理解决策理由
- 允许用户覆盖决策
- 持续优化 Soul profile 生成流程

**监控**: 记录用户覆盖决策的频率，分析原因

### 权衡: 灵活性 vs 简单性

**选择**: 优先保证简单性，预留扩展点

**理由**:
- 当前系统规模不需要过度灵活
- 简单的架构更易于理解和维护
- 预留扩展点，未来可以按需扩展

**优势**: 降低学习成本，减少维护负担

**劣势**: 可能需要重构才能支持某些高级场景

### 权衡: 性能 vs 质量

**选择**: 牺牲一定性能换取质量

**理由**:
- Decision Agent 的思考过程需要额外的 LLM 调用
- 但能做出更好的决策，减少返工
- 长期来看节省时间

**优势**: 提升决策质量，减少错误

**劣势**: 执行时间增加 10-20%

## Migration Plan

### 阶段 1: 实现新架构（不影响现有用户）

1. 创建 Agent 基类和接口
2. 实现 Decision Agent
3. 实现 Worker Agent
4. 添加单元测试和集成测试

### 阶段 2: 提供兼容层

1. 在 AutonomousExecutor 中添加 `legacy_mode` 参数
2. 默认使用新架构，但支持切换到旧流程
3. 更新文档，说明迁移路径

### 阶段 3: 逐步迁移用户

1. 发布新版本，标记旧 API 为 deprecated
2. 提供迁移指南和示例
3. 收集用户反馈，修复问题

### 阶段 4: 移除旧代码

1. 在下一个主版本中移除旧流程
2. 清理兼容层代码
3. 更新所有文档

### 回滚策略

如果新架构出现严重问题：
1. 立即切换到 `legacy_mode=True`
2. 发布 hotfix 版本
3. 分析问题原因，修复后重新发布

## Open Questions

1. **Decision Agent 的决策深度如何控制？**
   - 是否需要配置选项让用户选择"快速模式"vs"深度思考模式"？
   - 如何平衡决策质量和执行速度？

2. **Worker 输出选项时，Decision Agent 如何处理？**
   - 是否需要额外的 LLM 调用来理解选项？
   - 如何确保决策基于 Soul profile 而非随机选择？

3. **上下文管理的粒度如何确定？**
   - 需要保留多少历史决策？
   - 如何处理上下文过大导致的性能问题？

4. **如何评估 Decision Agent 的决策质量？**
   - 需要哪些指标？
   - 如何收集用户反馈？

这些问题将在实现过程中逐步解决，可能需要根据实际使用情况调整设计。
