## Context

当前系统由两个独立的 Python 脚本组成：
- `cli.py`: 通过 6 个问题生成 soul.md 和 soul.meta.yaml
- `chat.py`: 基于 soul.md 进行对话，支持 Ollama 和 Claude 后端

这两个脚本各自独立运行，缺少统一的工作流。用户需要先运行 cli.py 生成 soul，然后手动运行 chat.py 进行对话。更重要的是，chat.py 在遇到需要用户决策的场景时，无法自动参考 soul 中的决策偏好，仍然需要用户手动介入。

**约束条件**:
- 必须保持向后兼容，现有的命令行参数和用法仍然有效
- soul.md 和 soul.meta.yaml 的格式不能改变
- 需要支持现有的 Ollama 和 Claude 后端

**利益相关者**:
- 终端用户：期望获得流畅的端到端体验
- 开发者：需要清晰的代码结构和可扩展的架构

## Goals / Non-Goals

**Goals:**
- 提供统一的 CLI 入口，整合题目引导、soul 确认和代理决策
- 实现交互式的题目引导流程，提升用户体验
- 实现 soul 确认机制，确保用户理解并认可生成的画像
- 实现基于 soul 的代理决策，减少人工介入
- 记录代理决策过程，提供可追溯性

**Non-Goals:**
- 不改变现有的 soul 生成算法和评分逻辑
- 不修改 soul.md 和 soul.meta.yaml 的文件格式
- 不实现多用户管理或权限控制
- 不实现 GUI 界面，仅保持命令行工具

## Decisions

### 1. CLI 框架选择：使用 Click + Questionary

**决定**: 使用 Click 作为 CLI 框架，Questionary 作为交互式提示库。

**理由**:
- Click 是 Python 生态中最流行的 CLI 框架，文档完善，社区活跃
- Questionary 提供了丰富的交互式提示（选择、确认、文本输入等），用户体验好
- 两者可以无缝集成，Click 处理命令和参数，Questionary 处理交互流程

**替代方案**:
- argparse + input(): 原生方案，但交互体验差，不支持复杂的选择界面
- Typer: 基于 Click 的现代封装，但需要额外学习成本，且对复杂交互支持有限
- prompt-toolkit: 功能强大但过于底层，需要更多代码实现相同功能

### 2. 架构设计：模块化的命令结构

**决定**: 采用 Click 的命令组（group）模式，将功能拆分为多个子命令。

**架构**:
```
cinder-cli (主命令)
├── init (初始化：题目引导 + soul 生成)
├── confirm (确认 soul)
├── chat (对话模式，支持代理决策)
├── decisions (查看决策日志)
└── config (配置管理)
```

**理由**:
- 模块化设计便于维护和扩展
- 用户可以单独运行某个步骤，也可以通过 `init` 完成完整流程
- 保持向后兼容，`chat` 命令可以复用现有的 chat.py 逻辑

### 3. Soul 确认流程：三阶段确认

**决定**: 实现三阶段确认流程：展示 → 解释 → 调整。

**流程**:
1. **展示阶段**: 显示生成的 soul.md 核心内容（特质、决策画像、行为准则）
2. **解释阶段**: 用通俗语言解释每个维度的含义和影响
3. **调整阶段**: 允许用户重新回答某个问题或调整 trait 分数

**理由**:
- 用户需要理解 soul 的含义才能做出明智的确认
- 提供调整机会可以修正不准确的地方
- 三阶段流程平衡了确认的严谨性和用户体验

### 4. 代理决策机制：基于规则的决策引擎

**决定**: 实现基于规则的决策引擎，而非完全依赖 LLM。

**架构**:
```
决策请求 → 规则匹配 → Soul 规则应用 → LLM 增强 → 决策输出
```

**规则类型**:
- **风险规则**: 根据 risk_tolerance 决定是否接受高风险选项
- **沟通规则**: 根据 communication_preferences 决定输出格式
- **边界规则**: 根据 boundary_reminders 决定是否需要人工确认

**理由**:
- 纯 LLM 方案难以保证决策的一致性和可解释性
- 基于规则的引擎可以精确控制决策边界
- LLM 用于增强决策的自然语言表达，而非核心决策逻辑

**替代方案**:
- 纯 LLM 方案: 将 soul 作为 system prompt，让 LLM 自行决策。风险：不可控、不可解释
- 纯规则方案: 完全基于 if-else 逻辑。缺点：灵活性差，难以处理复杂场景

### 5. 决策日志存储：SQLite + JSON

**决定**: 使用 SQLite 存储决策日志，决策详情使用 JSON 格式。

**数据模型**:
```sql
CREATE TABLE decisions (
    id INTEGER PRIMARY KEY,
    timestamp TEXT NOT NULL,
    context TEXT NOT NULL,          -- JSON: 决策上下文
    soul_rules TEXT NOT NULL,       -- JSON: 应用的 soul 规则
    decision TEXT NOT NULL,         -- JSON: 决策结果
    confidence REAL NOT NULL,       -- 置信度 0-1
    requires_human BOOLEAN NOT NULL -- 是否需要人工确认
);
```

**理由**:
- SQLite 轻量级，无需额外服务，适合单机 CLI 工具
- JSON 格式灵活，可以存储复杂的决策详情
- 便于后续查询和分析

**替代方案**:
- 纯文件存储（JSON/YAML）: 查询效率低，不便于聚合分析
- PostgreSQL/MySQL: 过于重量级，增加部署复杂度

### 6. 配置管理：YAML 配置文件

**决定**: 使用 YAML 格式的配置文件 `~/.cinder/config.yaml`。

**配置项**:
- 默认后端（Ollama/Claude）
- 模型参数
- 代理决策的边界规则
- 日志级别和路径

**理由**:
- YAML 人类可读，便于用户手动编辑
- 支持复杂的数据结构
- Python 有成熟的 PyYAML 库

## Risks / Trade-offs

### 风险 1: 代理决策可能偏离用户真实意愿

**风险描述**: 即使基于 soul，代理决策仍可能在某些场景下做出不符合用户真实意愿的决定。

**缓解措施**:
- 实现严格的决策边界规则，高风险决策必须人工确认
- 提供决策回滚机制
- 在决策日志中明确标注置信度，低置信度决策需要人工确认
- 允许用户调整 soul 或添加例外规则

### 风险 2: 交互式流程可能降低效率

**风险描述**: 对于熟练用户，交互式确认流程可能显得繁琐。

**缓解措施**:
- 提供 `--skip-confirmation` 参数跳过确认
- 提供 `--quick` 模式，使用默认值快速完成
- 保存用户偏好，下次自动应用

### 风险 3: 向后兼容性可能限制架构优化

**风险描述**: 为了保持向后兼容，可能无法采用最优的架构设计。

**缓解措施**:
- 将旧接口标记为 deprecated，但保持功能
- 提供迁移指南，鼓励用户使用新接口
- 在下一个 major 版本中移除旧接口

### 风险 4: SQLite 可能在大规模使用时性能下降

**风险描述**: 如果用户频繁使用代理决策，SQLite 数据库可能变得很大，影响查询性能。

**缓解措施**:
- 实现日志清理机制，定期归档旧日志
- 提供日志导出功能，允许用户备份和清理
- 如果未来需要，可以迁移到更强大的数据库

## Migration Plan

### 阶段 1: 实现核心功能（v2.0.0）
1. 实现新的 CLI 框架（基于 Click）
2. 实现题目引导和 soul 生成
3. 实现 soul 确认流程
4. 实现代理决策引擎
5. 实现决策日志记录

### 阶段 2: 整合现有功能（v2.1.0）
1. 将 chat.py 的功能整合到新的 CLI
2. 保持旧的 cli.py 和 chat.py 作为兼容层
3. 提供迁移文档

### 阶段 3: 优化和扩展（v2.2.0）
1. 优化交互体验
2. 添加更多决策规则
3. 实现决策统计分析功能

### 回滚策略
- 保留旧的 cli.py 和 chat.py，用户可以继续使用
- 提供配置选项切换新旧模式
- 如果新版本出现严重问题，可以快速回退到旧版本

## Open Questions

1. **代理决策的边界如何精确界定？**
   - 需要明确哪些决策可以完全代理，哪些必须人工确认
   - 是否需要实现"学习模式"，根据用户反馈调整边界？

2. **如何处理 soul 的动态更新？**
   - 用户的行为可能随时间变化，soul 是否需要定期更新？
   - 如何平衡 soul 的稳定性和适应性？

3. **多语言支持的优先级？**
   - 当前系统仅支持中文，是否需要支持英文或其他语言？
   - 如果支持，soul 的生成逻辑是否需要调整？

4. **是否需要实现云端同步？**
   - 用户可能需要在多台设备上使用，是否需要实现 soul 和决策日志的云端同步？
   - 这会增加复杂度和成本，需要权衡
