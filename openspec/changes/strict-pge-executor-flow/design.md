## Context

当前 executor 实现存在以下问题：

**当前状态**:
- Plan 阶段仅基于关键词匹配（如 "web", "api"），无语义理解
- Generation 一次性生成代码，无迭代优化机制
- Evaluation 与 Decision 混淆，Decision 在 Evaluation 完成前就开始
- 低质量代码也会被接受和执行

**约束**:
- 必须保持向后兼容，现有 API 不能破坏性变更
- 不能增加新的外部依赖
- 执行时间增加在可接受范围内（< 2x）

**利益相关者**:
- 开发者：需要更高质量的代码生成
- 用户：需要更可靠的执行结果

## Goals / Non-Goals

**Goals:**
- 严格分离 Plan → Generation → Evaluation → Decision 四个阶段
- 每个阶段都有明确的质量控制和验证机制
- 实现迭代式代码生成，确保代码达到质量阈值
- 增强 Evaluation 维度，包含代码质量和 Soul 一致性检查
- 在 Evaluation 通过后才进行 Soul-based Decision

**Non-Goals:**
- 不改变现有的 Soul 画像机制
- 不修改决策日志的数据结构
- 不优化 LLM 的推理速度
- 不实现并行任务执行

## Decisions

### 决策 1: Plan 阶段使用 LLM 进行语义理解

**选择**: 使用 LLM 分析目标语义，而非仅依赖关键词匹配

**理由**:
- 关键词匹配无法处理复杂或模糊的目标
- LLM 可以理解上下文和隐含需求
- 生成的计划更加合理和完整

**备选方案**:
- 继续使用启发式规则 + 规则引擎
- 拒绝原因：无法处理语义复杂性

### 决策 2: Generation 阶段实现迭代循环

**选择**: 实现最多 3 次迭代的生成-评估-改进循环

**理由**:
- 一次性生成难以保证质量
- 迭代可以逐步改进代码
- 3 次迭代在质量和时间之间取得平衡

**备选方案**:
- 单次生成 + 后处理优化
- 拒绝原因：无法从根本上提升质量

### 决策 3: Evaluation 在 Decision 之前完成

**选择**: Evaluation 必须完成并达到阈值后才进入 Decision 阶段

**理由**:
- 避免在质量未知时就做决策
- 确保 Decision 基于完整的质量信息
- 流程更加清晰和可追溯

**备选方案**:
- Evaluation 和 Decision 并行
- 拒绝原因：流程混乱，难以调试

### 决策 4: 使用现有的 iterative_refinement 方法

**选择**: 复用 ReflectionEngine.iterative_refinement() 方法

**理由**:
- 已有实现，减少开发工作量
- 经过测试，稳定可靠
- 符合 DRY 原则

**备选方案**:
- 重新实现新的迭代机制
- 拒绝原因：重复造轮子

## Risks / Trade-offs

**风险 1: 执行时间增加**
- 影响: 迭代生成可能导致执行时间增加 50-100%
- 缓解: 设置合理的质量阈值（0.7），避免过度迭代
- 监控: 添加执行时间日志，超过阈值时告警

**风险 2: LLM 理解偏差**
- 影响: Plan 阶段可能误解用户意图
- 缓解: 添加计划验证步骤，低质量时请求用户澄清
- 监控: 记录计划质量评分，定期分析

**风险 3: 迭代无法收敛**
- 影响: 代码质量可能在迭代中反而下降
- 缓解: 保留最佳版本，设置最大迭代次数
- 监控: 追踪每次迭代的质量评分

**权衡: 质量vs速度**
- 选择牺牲一定的速度换取质量
- 理由: 低质量的代码反而需要更多时间修复
- 优势: 长期来看节省维护成本
