# Soul 问卷设计说明

## 目标

这套问卷不是为了贴一个好看的性格标签，而是为了生成一份可供 agent 使用的 `soul.md`。

它的目标只有一个：用尽可能少、但信息密度足够高的问题，提取一个人在真实决策里的稳定偏好，并把这种偏好转成可执行的代理规则。

因此，这套题目关心的不是抽象的“你是什么人”，而是更接近实际决策的六件事：

- 面对未知时，你先建模、先试错、先看关系还是先守边界
- 面对计划被打断时，你的情绪如何波动、恢复速度如何、是否需要确认感
- 面对冲突时，你优先关系、真相、机制，还是推进速度
- 面对风险时，你偏确定性、平衡试错、冲上限，还是继续找兼顾解
- 面对疲惫时，你靠独处、熟人连接、新刺激还是自由流动恢复
- 面对长期受阻时，你靠纪律、适应、外部推动还是意义重连继续前进

这意味着它本质上是一套“决策人格建模”问卷，而不是传统心理测试。

## 为什么升级为 6 个问题

原始方案的 5 个问题已经足够形成一版可用画像，但还缺了一个对 agent 非常关键的维度：**情绪调节**。

agent 在真实陪伴和辅助决策中，遇到的并不总是“理性选择题”，更常见的是：

- 用户刚被打断计划，情绪在波动
- 用户担心连锁后果，需要稳定感
- 用户表面能继续推进，但内部已经在被消耗

如果没有这一维，agent 很容易在错误的时机给出过多分析，或者把本该先安抚的问题，当成纯理性优化问题处理。

因此现在的最优版本采用 6 题结构：

1. 未知应对
2. 情绪调节
3. 冲突处理
4. 风险偏好
5. 能量恢复
6. 长期动力

这六个维度合在一起，已经足以支持一个高可用的代理偏好建模。

## 核心哲学

### 1. 不测理想自我，测真实取舍

很多人格题直接问：

- 你是不是理性
- 你是不是内向
- 你是不是自律

这类问题很容易被“理想中的自己”污染。用户会回答自己希望成为什么样，而不是自己在关键时刻真正如何做决定。

所以这套问卷坚持用情境化、被迫取舍的问题。它不问“你觉得自己怎样”，而问“如果现在真的发生这件事，你最可能先怎么做”。

这背后的哲学是：

**人格在决策里的体现，通常比人格在自我叙述里的表达更稳定。**

### 2. 不追求完整人格，追求最小可用画像

完整人格极其复杂，不可能通过 6 个单选题被完整还原。

但 agent 并不需要一个人的完整心理史。它需要的是一组足够稳定、足够可操作的偏好参数，来帮助回答以下问题：

- 该给保守方案还是探索方案
- 该先安抚还是先分析
- 该讲逻辑还是先处理关系
- 该帮助行动，还是帮助澄清
- 该提醒风险，还是补足动力

因此这套问卷采用的是“最小可用人格模型”：

**问题少，但每题都直连一个高价值决策入口。**

### 3. 不输出标签，输出规则

传统人格测评喜欢给出某个类型标签。

但对 agent 来说，标签用处有限。agent 真正需要的是规则，例如：

- 这个用户面对不可逆风险时应提高谨慎权重
- 这个用户在计划被打断时需要先降低复杂度
- 这个用户更相信逻辑证据而不是情绪鼓动
- 这个用户的长期执行依赖意义而不是纪律

所以 `soul.md` 的核心不是“你是谁”，而是：

**当 agent 替你做分析、建议和排序时，应该优先保护什么，避免什么，如何补偿你的典型盲点。**

## 理论基础

这套设计不是照搬某一个经典模型，而是面向 agent 场景做的组合建模。

### 大五人格的启发

它借用了大五人格里最有操作价值的方向：

- 开放性：是否偏探索、新刺激和试错
- 尽责性：是否偏结构、纪律、持续推进
- 外向性：能量恢复更来自外界互动还是独处
- 宜人性：在冲突中是否更优先关系稳定
- 情绪稳定性：是否容易被扰动影响，恢复速度如何

### 行为决策理论的启发

它吸收了现实决策中更关键的变量：

- 风险偏好
- 损失规避
- 可逆性判断
- 不确定性下的起手策略

这些因素对 agent 的建议风格，比一个人格标签更直接。

### 动机理论的启发

第六题直接对应长期推进系统，它区分：

- 纪律驱动
- 意义驱动
- 外部推动
- 动态适应

这比简单区分“自律 / 不自律”更接近真实执行机制。

### 认知风格与情绪调节的启发

整套题目同时区分：

- 证据导向
- 关系导向
- 行动导向
- 结构导向
- 探索导向
- 情绪反应强度
- 情绪恢复速度
- 外部确认需求

这正是 agent 组织建议、判断节奏和控制复杂度时最需要的输入。

## 六道题分别在测什么

## Q1 未知应对

这一题测的是进入未知局面的第一反应。

它区分的是四种起手式：

- 建框架
- 先试错
- 先看关系和现实约束
- 先划风险边界

它决定 agent 面对模糊任务时，应该先帮用户澄清结构，还是先建议一个小实验。

## Q2 情绪调节

这一题测的是计划被打断时的波动方式。

它不只是测“会不会生气”，而是在拆三个关键能力：

- 情绪反应强度
- 恢复速度
- 是否需要外部确认感

这一题对 agent 特别重要，因为它直接影响：

- 现在是不是适合讲很多分析
- 是否应该先减负、先给锚点
- 是否要先稳定状态，再讨论决策本身

## Q3 冲突处理

这一题测的是关系压力下的优先排序。

它区分的是：

- 关系优先
- 真相优先
- 机制优先
- 推进优先

这决定了 agent 在协作、分歧、谈判场景中应该怎样组织建议与措辞。

## Q4 风险偏好

这一题测的是对确定性和上限的排序方式。

它会告诉 agent：

- 应该保守一点
- 应该稳健主策略 + 局部探索
- 应该激进冲高上限
- 还是先暂停，继续优化方案空间

这直接决定建议的默认风险姿态。

## Q5 能量恢复

这一题测的是恢复机制，而不只是内向外向。

它区分的是：

- 独处沉浸恢复
- 熟人连接恢复
- 新刺激恢复
- 自由流动恢复

这会影响 agent 在用户疲惫、过载、低状态时的支持方式。

## Q6 长期动力

这一题测的是人在长期受阻时靠什么重启。

它区分的是：

- 纪律和拆解
- 路径重构
- 外部监督
- 意义重连

这决定了 agent 在用户卡住时，应该帮他做计划、换路径、找外部约束，还是先找回意义。

## 从答案到画像的推理方式

这套问卷不是把每个选项直接贴成一个标签，而是采用两层推理。

### 第一层：选项映射到 trait 向量

每个选项会作用到一组连续变量，例如：

- `structure`
- `exploration`
- `risk_tolerance`
- `evidence_orientation`
- `relationship_orientation`
- `action_bias`
- `social_energy`
- `meaning_drive`
- `discipline_drive`
- `adaptability`
- `emotional_reactivity`
- `recovery_speed`
- `reassurance_need`

系统从中性基线出发，再根据每题答案叠加增减分。

这意味着结果不是非黑即白，而是连续谱。

### 第二层：分数组合转成可执行规则

单一维度本身意义有限，更重要的是组合关系。

例如：

- 高结构 + 低行动偏好，意味着先建模后行动
- 高情绪反应 + 高确认需求，意味着先稳状态再做选择
- 高证据导向 + 低关系导向，意味着逻辑型说服更有效
- 高意义驱动 + 低纪律驱动，意味着状态低落时要先重连意义
- 低社交能量，意味着不要默认推荐高强度社交解法

因此最终输出的是 agent 可执行的规则，而不是一段空泛结论。

## 为什么输出拆成 soul.md 和 soul.meta.yaml

为了兼顾“人类可读”和“程序可读”，现在输出被拆成两个文件：

- `soul.md`：只保留纯 Markdown 正文，适合阅读、预览和分享
- `soul.meta.yaml`：只保留结构化元数据，适合程序、agent 和工作流直接解析

这样拆分的原因是，很多 Markdown 预览器对 YAML front matter 的兼容并不一致，容易出现显示异常、样式混乱或阅读干扰。

拆分后，职责更清晰：

### 机器可读层

`soul.meta.yaml` 用于程序消费，例如：

- 原始问答
- trait 向量
- 核心特质
- 决策画像
- agent 指令偏好

### 代理可执行层

`soul.md` 中的自然语言模块告诉 agent：

- 该如何组织建议
- 该优先保护什么
- 该避免哪些决策方式
- 该怎样做场景适配
- 该提醒哪些边界
- 该如何动态更新画像

因此，`soul.md` 负责表达，`soul.meta.yaml` 负责结构，二者共同构成完整画像。

## 为什么 soul.md 要包含更多落地模块

一个好的代理画像不能只停留在“这个人偏谨慎、偏逻辑”。

真正有用的 `soul.md` 还需要把人格翻译成可落地的模块：

- 决策核心准则
- 决策禁忌
- 典型场景适配
- 高概率盲点
- 反向补偿策略
- 决策边界提醒
- 动态调整机制

这些模块的作用是把性格偏好直接转成 agent 的行为约束。

## 这套方法的边界

这套方案很实用，但必须清楚它的边界。

### 它不是临床心理工具

它不能用于心理诊断、病理评估或深层人格治疗场景。

### 它描述的是偏好，不是命运

结果反映的是相对稳定的决策倾向，而不是绝对真理。
人在不同阶段会变化，所以 `soul.md` 应该允许更新。

### 它适合做代理校准，不适合做绝对定义

它最适合回答的是：

- agent 应该默认什么建议风格
- agent 该优先补哪类短板
- agent 什么时候应提醒风险、什么时候应降低复杂度

它不适合回答的是：

- 这个人本质上到底是什么样的人
- 某种人格是不是优于另一种人格

## 使用建议

如果把它用于真实 agent，建议遵循以下原则：

- 把当前结果当作初始画像，而不是最终真相
- 保留原始答案和原因，便于增量修正
- 后续允许通过新问答和真实行为持续更新 `soul.md`
- 当短期情绪与长期人格冲突时，不机械压制，而是并列展示两种方案及其代价
- 优先把这套系统用于建议排序、沟通风格、风险提示和状态管理，而不是替用户硬做最终决定

## 一句话总结

这套问卷的哲学可以概括为一句话：

**不是用 6 个问题定义一个人，而是用 6 个高信息密度的取舍场景，逼近一个人在关键决策中的稳定偏好，并把这种偏好转译成 agent 可执行的行为规则。**
